# Report Specification (Step D)

**Status:** Draft for ratification
**Version:** 0.1.0-draft (§5 finding-order key corrected 2026-06-09 — see §5 note)
**Depends on:** `INPUT_OUTPUT_SPEC.md` §5 (output contract), `schemas/finding.schema.json`, the eight rule cards (`rules/toa/`), and the active court profile.

This document is normative for the Markdown report citetab writes beside
the regenerated `.docx`. It pairs with `finding.schema.json`: the schema
defines what a finding *is*; this spec defines how a run's TOA and findings
are *rendered* for a human. The three files in `examples/reports/` are
instances of this spec and are regenerated by the build.

## 1. The file

- Default name `{input_stem}.toa-report.md`, written alongside the
  regenerated `.docx` (`INPUT_OUTPUT_SPEC.md` §5.2). `--output` does not
  rename it independently in v1; it tracks the `.docx` stem.
- Always written — even when the `.docx` is suppressed (TT-005). When the
  `.docx` is suppressed, the report is the *only* output and still carries
  the full generated TOA.
- UTF-8, LF line endings, terminal newline. Byte-stable for a given input
  and environment, with two exceptions named in §6.

## 2. Order: generation first, diagnostics second

The report has exactly three parts, in this order:

1. **Run header** (§3) — what ran, against what, with what result.
2. **Table of Authorities** (§4) — the generated table, readable.
3. **Findings** (§5) — diagnostics, most actionable first.

This order is the product posture in `LIABILITY.md`: the table is the
product; findings are disclosures around it. The TOA always precedes the
findings, even when an error finding is present.

## 3. Run header

A level-1 title followed by a fixed metadata block:

```
# citetab report — {input_filename}

- engine {engine_version} · rule pack toa {rule_pack_version} · profile {profile_id} {profile_version}
- input: {input_filename} (sha256 {first 12 hex}…)
- render: {render_engine} {render_version} · fonts {ok | substituted ({pairs})}
- placement: {heading "<text>" | [[TOA]] marker, consumed | --toa-heading "<text>"}
- pagination: converged in {n} iteration(s) (cap 5) | DID NOT CONVERGE (cap 5 reached)
- generated: {ISO-8601 UTC}
- findings: {e} error · {w} warning · {i} info
- output: {written {stem}.toa.docx | SUPPRESSED — no placement found}
```

The header records the four version tracks that produced the run (engine,
rule pack, profile, plus each finding's rule version in §5) so any report
is reproducible against the exact definitions in force.

## 4. Table of Authorities

A level-2 `## Table of Authorities` heading, then one level-3 subsection
per profile group in the profile's group order (FRAP: Cases, Statutes,
Regulations, Rules — empty groups are omitted). Within a group, entries are
in the profile's sort order. Each entry is one line:

```
{authority as the profile formats it}{tab}{page list | passim}
```

- Page lists are ascending, de-duplicated, comma-separated: `2, 3, 5`.
- An authority cited on more than the profile's passim threshold (FRAP:
  **5**, so **6+** pages) renders `passim` instead of a page list. Exactly
  5 pages lists the pages — the threshold is the boundary, not the trigger.
- The rendered table is **semantically identical** to the TOA written into
  the `.docx`; the report never shows a table the document doesn't contain.

## 5. Findings

A level-2 `## Findings` heading. If there are none: the single line
`No findings.` Otherwise, findings are ordered **error → warning → info**,
and within a severity by **rule id**, then by first evidence location
(document order). Each finding renders as:

```
### {rule_id} · {rule name} — {severity} ({confidence})
{message}

- subject: {subject}                      (omitted when the finding has none)
- evidence: {rendered occurrences / entries / computed values}
- authority: {source section[; source section …]}
- before filing: {remediation_hint}       (omitted when the finding has none)
```

Rendering conventions:

- **Occurrence evidence** renders as `¶{paragraph_index} p.{page} "{excerpt}"`,
  one per line or semicolon-joined when short. `page` renders `p.?` when null
  (finding fired before a stable render).
- **Input-TOA entry evidence** renders the `verbatim_text` quoted exactly.
- **Computed values** render as a compact `key: value` list or a small
  table when the value is a mapping (e.g. TT-003's per-entry corrections).
- The `finding_id` ULID and `evaluated_at_utc` are **not** rendered in the
  human report (see §6); they live in the optional sidecar (§7).

### Per-rule emission and aggregation

- **One finding per subject** for the per-authority rules: TT-001
  (each orphan short form), TT-002 (each authority added), TT-004 (each
  phantom entry).
- **One aggregate finding** for TT-003 (stale page numbers): a single
  info finding whose `computed_values` maps each corrected entry to its
  input pages and corrected pages. Stale numbers are a property of the
  input TOA as a whole, and the tool's job is to correct them; the single
  finding discloses the correction rather than fragmenting it.
- **One run-level finding** for TT-005, TT-006, TT-007, TT-008 (no
  subject).

### Severity is an instruction, not a verdict

Per `AI_GUARDRAILS.md` §5, a finding states what the tool did or what the
user must do — never that the brief is correct. `error` = the user must
act; `warning` = verify before filing; `info` = a disclosure or a
correction the tool made. **TT-008 renders at `warning`** per its rule
card; `INPUT_OUTPUT_SPEC.md` §3.1 still says "info" and is to be patched on
next touch — the card governs.

### Note — finding-order reconciliation (2026-06-09)

An earlier draft of this section ordered findings within a severity by
document order, then rule id. That conflicted with `PRD.md` §11 FR-10 and
`rules/README.md` ("Cross-rule concerns"), both of which specify **rule id,
then document order**. The two PRD/rule-pack sources govern: this section
was corrected to **severity → rule id → document order** so all three
agree. The change affects only the tie-break between two findings of the
same severity, and none of the committed `examples/reports/` instances
contain two same-severity findings, so they are unaffected.

## 6. Determinism

Given the same input file and the same environment (same profile, rule
pack, fonts, and LibreOffice version), the report is byte-for-byte
identical across runs, with two exceptions, both confined to the header:
the `generated:` timestamp and — only if a future version surfaces it —
any ULID. Tests normalize the `generated:` line before comparing against
`examples/reports/`. Running citetab on its own output reproduces the same
TOA with zero changes (`INPUT_OUTPUT_SPEC.md` §3.3); the second run's
report differs only in the normalized timestamp.

Font substitution is part of "environment": the committed example reports
were generated under LibreOffice with Liberation substitution, so each
carries a TT-008 warning. In a Times New Roman environment the same inputs
produce the same tables with no TT-008 finding.

## 7. Machine-readable sidecar (deferred)

`INPUT_OUTPUT_SPEC.md` §5.3 leaves a serialized sidecar (registry +
findings as JSON, including the ULIDs and timestamps omitted from §5) as an
open question whose only identified driver is idempotency testing. It is
**not** part of v1 output unless the test suite demonstrates it is needed;
if added, it is `{input_stem}.toa-report.json` and validates each finding
against `finding.schema.json`. The human Markdown report remains primary.
